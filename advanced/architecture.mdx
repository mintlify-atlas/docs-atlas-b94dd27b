```text

```text

## Server Architecture

The `packages/server/` package provides three server modes:

### Plan Server (`index.ts`)

Handles plan review with approval/denial flow.

**Key endpoints:**
- `GET /api/plan` - Returns plan with metadata
- `POST /api/approve` - Approve plan
- `POST /api/deny` - Deny with feedback
- `GET /api/plan/version` - Fetch specific version
- `GET /api/plan/versions` - List all versions
- `GET /api/plan/history` - List all plans in project
- `POST /api/plan/vscode-diff` - Open diff in VS Code

**Features:**
- Version history (packages/server/storage.ts:29)
- Plan diffing (packages/ui/utils/planDiffEngine.ts)
- Integration with Obsidian/Bear (packages/server/integrations.ts:18)
- Image upload/serving (packages/server/index.ts:296)
- Agent switching (OpenCode only)

### Review Server (`review.ts`)

Handles code review for git diffs.

**Key endpoints:**
- `GET /api/diff` - Returns git diff patch
- `POST /api/feedback` - Submit review feedback
- `GET /api/image` - Serve images
- `POST /api/upload` - Upload images

**Features:**
- Git diff parsing
- File tree navigation
- Inline annotations
- Syntax highlighting

### Annotate Server (`annotate.ts`)

Handles annotation of arbitrary markdown files.

**Key endpoints:**
- `GET /api/plan` - Returns markdown content with mode flag
- `POST /api/feedback` - Submit annotations
- `GET /api/image` - Serve images
- `POST /api/upload` - Upload images

**Features:**
- Markdown file loading
- Same annotation UI as plan review
- Feedback routing to agent

## Data Flow

### Plan Review Flow

```text
1. Agent exits plan mode
   ↓
2. Hook/tool intercepts
   ↓
3. Server saves to history
   ~/.plannotator/history/{project}/{slug}/001.md
   ↓
4. Server detects previous version (if exists)
   ↓
5. Server starts on random/fixed port
   ↓
6. Browser opens plan viewer
   ↓
7. UI fetches plan + metadata
   GET /api/plan
   {
     plan: "# My Plan...",
     previousPlan: "# My Plan (old)...",
     versionInfo: { version: 2, totalVersions: 2 }
   }
   ↓
8. UI parses markdown → blocks
   parser.ts:parseMarkdownToBlocks()
   ↓
9. User adds annotations
   Annotation stored in React state
   ↓
10. User clicks Approve/Deny
    ↓
11. POST /api/approve or /api/deny
    { feedback: "exported annotations" }
    ↓
12. Server saves decision snapshot (if enabled)
    ~/.plannotator/plans/{slug}-approved.md
    ↓
13. Server resolves promise, returns to plugin
    ↓
14. Plugin outputs decision to stdout/agent
```text

### Version History Flow

```text
1. Plan arrives at server
   ↓
2. Generate slug from heading + date
   storage.ts:generateSlug(plan)
   "my-feature-2026-03-03"
   ↓
3. Check for existing versions
   ~/.plannotator/history/{project}/{slug}/
   ↓
4. Compare with latest version
   If identical → skip save (deduplication)
   If different → save as next version
   ↓
5. Increment version number
   001.md, 002.md, 003.md...
   ↓
6. Server sends previousPlan to UI
   ↓
7. UI computes diff automatically
   planDiffEngine.ts:computeDiff()
   ↓
8. User can select any version from sidebar
   Version Browser → fetch version → compute diff
```text

### URL Sharing Flow

```text
1. User clicks "Share" button
   ↓
2. UI compresses plan + annotations
   sharing.ts:compressAndEncode()
   JSON → deflate-raw → base64 → URL-safe
   ↓
3. If size < 2000 chars → hash URL
   https://share.plannotator.ai/#<data>
   ↓
4. If size > 2000 chars → short URL
   User confirms "Create short URL?"
   ↓
5. POST to paste service
   POST /api/paste
   { data: "compressed..." }
   Returns: { id: "xJ9k2mP1" }
   ↓
6. Generate short URL
   https://share.plannotator.ai/p/xJ9k2mP1
   ↓
7. Portal loads URL
   If /p/:id → fetch from paste service
   If /#<data> → decode from hash
   ↓
8. Decompress plan + annotations
   ↓
9. Render plan with annotations applied
```text

## Paste Service Architecture

Platform-agnostic core with pluggable storage backends.

```text
apps/paste-service/
├── core/
│   ├── handler.ts               # HTTP request handler
│   ├── storage.ts               # PasteStore interface
│   └── cors.ts                  # CORS headers
├── stores/
│   ├── fs.ts                    # Filesystem backend (Bun)
│   ├── kv.ts                    # Cloudflare KV backend
│   └── s3.ts                    # S3-compatible backend
└── targets/
    ├── bun.ts                   # Bun runtime entry
    └── cloudflare.ts            # Cloudflare Worker entry
```text

**Abstraction pattern:**
```typescript
interface PasteStore {
  put(id: string, data: string, ttlSeconds: number): Promise<void>;
  get(id: string): Promise<string | null>;
}
```text

Each target wires up its storage backend:
```typescript
// Bun
const store = new FsPasteStore(dataDir);
handleRequest(request, store, cors, { maxSize, ttlSeconds });

// Cloudflare
const store = new KvPasteStore(env.PASTE_KV);
handleRequest(request, store, cors);
```text

## Security Model

### Image Upload Security

**Validation:** packages/server/index.ts:296

- File extension whitelist (png, jpg, jpeg, gif, svg, webp)
- Stored in temp directory with random UUID names
- Path traversal protection (resolved paths must stay within temp dir)

## Extension Points

Plannotator is designed for extensibility:

### Add Storage Backend

1. Implement `PasteStore` interface
2. Create new file in `apps/paste-service/stores/`
3. Wire up in target file

### Add Integration

1. Implement save function in `packages/server/integrations.ts`
2. Add API endpoint in `packages/server/index.ts`
3. Add UI in `packages/ui/components/Settings.tsx`

### Add Server Mode

1. Create new server file in `packages/server/`
2. Follow pattern from `review.ts` or `annotate.ts`
3. Add subcommand to plugin entry point

## Performance Considerations

### Build Optimization

- Single-file HTML bundles eliminate HTTP requests
- Vite code splitting for large dependencies
- CSS inlined in production builds

### Runtime Optimization

- Random ports avoid conflicts
- Port retry logic (5 attempts with 500ms delay)
- Plan deduplication (skip identical versions)
- Paste expiration sweep on startup only

### Compression

URL sharing uses deflate-raw compression:

```typescript
// 50 KB plan → 5 KB compressed (typical 10:1 ratio)
compress(plan) → deflate-raw → base64 → URL-safe
```text
